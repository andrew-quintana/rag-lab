# RFC 001 — RAG Lab (RAG System Testing Suite)

## Context

This RFC defines the technical architecture and implementation plan for the RAG Lab system, as specified in [PRD001.md](./PRD001.md). The RAG Lab is a research and development tool that enables AI engineers to test and iterate on RAG (Retrieval-Augmented Generation) system components through a simple, modular pipeline.

**Status**: The RAG pipeline is currently not implemented. This RFC provides the technical design for implementing the complete system.

## Design Overview

The RAG Lab consists of two distinct, serialized pipelines that run entirely on the same server:

1. **Upload Pipeline (`POST /upload`)**: Processes documents and builds the retrieval index
2. **Query Pipeline (`POST /query`)**: Performs RAG retrieval and LLM generation

Both pipelines are designed for:
- **Determinism**: Reproducible results for testing and validation
- **Modularity**: Isolated components that can be tested independently
- **Simplicity**: Minimal abstractions, transparent components
- **Extensibility**: Easy to swap components (chunking strategies, embedding models, prompt versions)

## Interface Contracts (verbatim)

### Python API Interface

```python
def run_rag(query: Query, prompt_version: str = "v1", config: Optional[Config] = None) -> ModelAnswer:
    """
    Run the complete RAG pipeline: query embedding → retrieval → generation → logging.
    
    Args:
        query: Query object with text and optional metadata
        prompt_version: Version of the prompt to use (default: "v1")
        config: Application configuration (optional, defaults to Config.from_env())
        
    Returns:
        ModelAnswer object containing:
        - text: Generated answer
        - query_id: Unique query identifier
        - prompt_version: Prompt version used
        - retrieved_chunk_ids: List of chunk IDs used in generation
        - timestamp: Generation timestamp
        
    Raises:
        AzureServiceError: If Azure service calls fail
        ValueError: If query is invalid
        ValueError: If prompt version not found in database
    """
```

## Implementation Plan

### Phase 1: Embedding Generation

**Component**: `rag_eval/services/rag/embeddings.py`

**Implementation Details:**
- Use Azure AI Foundry embedding API (OpenAI-compatible)
- Support batch embedding generation for efficiency
- Use same embedding model for both chunks and queries (enforced via config)
- Implement retry logic with exponential backoff
- Validate embedding dimensions match expected model output

**Key Functions:**
```python
def generate_embeddings(chunks: List[Chunk], config) -> List[List[float]]
def generate_query_embedding(query: Query, config) -> List[float]
```

**Dependencies:**
- Azure AI Foundry endpoint and API key (from Config)
- Embedding model name (default: "text-embedding-3-small")

### Phase 2: Azure AI Search Integration

**Component**: `rag_eval/services/rag/search.py`

**Implementation Details:**
- Automatically create Azure AI Search index with vector field (idempotent - check if exists first)
- Use lightweight schema with minimal required fields
- Index chunks with embeddings and metadata
- Implement vector similarity search (cosine similarity)
- Support top-k retrieval with configurable k (default: 5)
- Handle empty index gracefully
- Never perform destructive resets (preserve existing index data)

**Key Functions:**
```python
def index_chunks(chunks: List[Chunk], embeddings: List[List[float]], config) -> None
def retrieve_chunks(query: Query, top_k: int = 5, config=None) -> List[RetrievalResult]
```

**Index Schema:**
- `id`: Chunk ID (string, key)
- `chunk_text`: Chunk content (string, searchable)
- `embedding`: Vector embedding (Collection(Edm.Single), vectorizable)
- `document_id`: Source document ID (string)
- `metadata`: Additional metadata (JSON)

**Dependencies:**
- Azure AI Search endpoint, API key, index name (from Config)
- Index will be automatically created on first use with idempotent checks

### Phase 3: Prompt Template System

**Component**: `rag_eval/services/rag/generation.py` (prompt loading)

**Implementation Details:**
- Load prompt templates from Supabase `prompt_versions` table
- Query by `version_name` (e.g., "v1", "v2") to retrieve `prompt_text`
- Template format: Text with placeholders for `{query}` and `{context}`
- Validate prompt version exists in database before use
- Cache loaded prompts in memory for performance (avoid repeated DB queries)
- Handle missing prompt versions with clear error messages

**Prompt Template Format:**
```
# System Instructions
You are a helpful assistant that answers questions based on the provided context.

# Context
{context}

# User Query
{query}

# Instructions
Answer the query using only the information from the context above.
```

**Database Schema:**
- Table: `prompt_versions`
- Fields: `version_id` (PK), `version_name` (unique), `prompt_text` (TEXT), `created_at`
- Reference: `model_answers.prompt_version` → `prompt_versions.version_name`

**Key Functions:**
```python
def load_prompt_template(version: str, query_executor: QueryExecutor) -> str
def construct_prompt(query: Query, retrieved_chunks: List[RetrievalResult], prompt_version: str, query_executor: QueryExecutor) -> str
```

**Dependencies:**
- Supabase connection (from Config)
- `QueryExecutor` from `rag_eval/db/queries.py`
- `prompt_versions` table must exist (already in schema)

### Phase 4: LLM Answer Generation

**Component**: `rag_eval/services/rag/generation.py`

**Implementation Details:**
- Use Azure AI Foundry (OpenAI-compatible API) for generation
- Support GPT-4o-mini (default) and GPT-4o models
- Configure temperature for deterministic behavior (temperature=0.1)
- Implement retry logic with exponential backoff
- Parse and validate LLM responses

**Key Functions:**
```python

    query: Query,
    retrieved_chunks: List[RetrievalResult],
    prompt_version: str,
    config
) -> ModelAnswer
```

**Generation Parameters:**
- Model: Configurable via `AZURE_AI_FOUNDRY_GENERATION_MODEL` (default: "gpt-4o")
- Temperature: 0.1 (for reproducibility)
- Max tokens: 1000 (configurable)
- System prompt: From `prompt_versions` table in Supabase

**Dependencies:**
- Azure AI Foundry endpoint and API key (from Config)
- Prompt template system (Phase 3)

### Phase 5: Pipeline Orchestration

**Component**: `rag_eval/services/rag/pipeline.py`

**Implementation Details:**
- Orchestrate complete query pipeline flow
- Generate query IDs for traceability
- Handle errors gracefully with proper logging
- Measure and log latency metrics
- Ensure deterministic execution order

**Pipeline Flow:**
1. Generate query ID and timestamp
2. Embed query using same model as chunks
3. Retrieve top-k chunks from Azure AI Search
4. Load prompt template for specified version
5. Construct prompt with query and retrieved context
6. Generate answer using Azure AI Foundry
7. Assemble ModelAnswer with metadata
8. Log to Supabase (Phase 6)

**Key Functions:**
```python
def run_rag(query: Query, prompt_version: str = "v1", config: Optional[Config] = None) -> ModelAnswer
```

**Dependencies:**
- All previous phases (embeddings, search, generation)
- Query ID generation utility

### Phase 6: Supabase Logging

## Technical Decisions

### Decision 1: Deterministic Chunking Strategy

**Decision**: Use fixed-size chunking as the default, with LLM-based chunking as an optional fallback.

**Rationale**:
- Fixed-size chunking is deterministic and reproducible (critical for testing)
- LLM-based chunking introduces non-determinism and additional cost
- Fixed-size chunking is simpler and faster
- Can be extended later with configurable chunk sizes

**Implementation**: The `chunk_text()` function defaults to `use_llm=False`, using `chunk_text_fixed_size()` with configurable `chunk_size` (default: 1000) and `overlap` (default: 200).

### Decision 2: Single Embedding Model for Chunks and Queries

**Decision**: Enforce that chunks and queries use the same embedding model (OpenAI's text-embedding-3-small).

**Rationale**:
- Vector similarity search requires embeddings in the same vector space
- Using different models would produce meaningless similarity scores
- Simplifies configuration and reduces errors

**Implementation**: Both `generate_embeddings()` and `generate_query_embedding()` use the same model from `config.azure_ai_foundry_embedding_model`.

### Decision 3: Prompt Template Versioning via Database

**Decision**: Store prompt templates in Supabase Postgres `prompt_versions` table.

**Rationale**:
- Centralized management of prompt versions
- Easy to update prompts without code deployments
- Supports A/B testing by switching versions dynamically
- Consistent with other system data (queries, answers, evaluations)
- Database already has `prompt_versions` table with proper schema
- Enables version tracking and audit trail via `created_at` timestamp

**Implementation**: Templates stored in `prompt_versions` table with `version_name` (e.g., "v1", "v2") and `prompt_text` fields. Loaded on-demand from database and cached in memory for performance.

### Decision 4: Serialized, Synchronous Processing

**Decision**: All pipeline operations run synchronously in a single process.

**Rationale**:
- Simpler architecture (no async complexity)
- Easier to debug and test
- Sufficient for R&D/testing use case
- Deterministic execution order

**Implementation**: All functions are synchronous, FastAPI endpoints use `async def` but call synchronous pipeline functions.

### Decision 5: Local Logging for Upload Pipeline

**Decision**: Upload pipeline events are logged locally, not to Supabase.

**Rationale**:
- Upload pipeline is for indexing, not query evaluation
- Reduces database load
- Supabase is reserved for evaluation/telemetry data
- Local logs sufficient for debugging

**Implementation**: Upload pipeline uses standard Python logging, not Supabase integration.

### Decision 6: Top-K Retrieval Default

**Decision**: Default to top-5 chunks for retrieval (`top_k=5`).

**Rationale**:
- Balance between context richness and prompt length
- Configurable for experimentation
- Common practice in RAG systems
- Sufficient for most queries

**Implementation**: `retrieve_chunks()` accepts `top_k` parameter with default value of 5.

### Decision 7: Error Handling Strategy

**Decision**: Fail fast with clear error messages, implement retry logic for Azure service calls.

**Rationale**:
- Clear errors help debugging during R&D
- Retry logic handles transient Azure service failures
- Fail fast prevents partial/corrupted state

**Implementation**:
- Retry logic with exponential backoff for Azure API calls (3 retries max)
- Clear error messages with context
- HTTP status codes: 400 (bad request), 500 (server error), 501 (not implemented)

## Package Structure Standards

### Overview

All packages in the `rag_eval` codebase must follow standardized package structure conventions to ensure proper imports, testability, and maintainability.

### Requirements

#### 1. `__init__.py` Files

- **All packages must have `__init__.py` files**: Every directory that is a Python package must contain an `__init__.py` file, even if it's empty (though it should contain exports).

- **Export Pattern**: Each `__init__.py` file should:
  - Include a docstring describing the package/module purpose
  - Export commonly used classes/functions using `from ... import ...` statements
  - Define an `__all__` list for explicit exports
  - Follow this pattern:
    ```python
    """[Package description]"""
    
    from rag_eval.[module].[submodule] import Class1, Class2, function1
    
    __all__ = [
        "Class1",
        "Class2", 
        "function1",
    ]
    ```

#### 2. Import Conventions

- **Absolute Imports Preferred**: Always use absolute imports from the package root:
  ```python
  from rag_eval.core.interfaces import Chunk
  from rag_eval.services.rag.chunking import chunk_text
  ```
  
- **No `sys.path` Manipulation**: Production code must not manipulate `sys.path`. This is only acceptable in test utilities if absolutely necessary, and should be clearly documented.

- **Module Execution**: Modules should be runnable as Python modules:
  ```bash
  python -m rag_eval.services.rag.chunking
  ```

#### 3. Package Organization

- **Core Primitives** (`rag_eval/core/`): Interfaces, exceptions, configuration, logging
- **Services** (`rag_eval/services/`): Business logic organized by domain (rag, evaluator, meta_eval)
- **Database** (`rag_eval/db/`): Database connection, queries, models
- **API** (`rag_eval/api/`): FastAPI application and routes
- **Utils** (`rag_eval/utils/`): Utility functions

#### 4. Testing Configuration

- **pytest Configuration**: The `pyproject.toml` must include:
  ```toml
  [tool.pytest.ini_options]
  pythonpath = ["."]  # Ensures pytest can find rag_eval package
  ```
  This allows pytest to resolve `rag_eval.*` imports when run from the `backend/` directory.

#### 5. Validation Checklist

When implementing new packages or modules, verify:
- [ ] Package has proper `__init__.py` file with exports
- [ ] Imports work correctly (`from rag_eval.* import ...`)
- [ ] Module can be run as `python -m rag_eval.*.*`
- [ ] No `sys.path` manipulation in production code
- [ ] pytest can discover and run tests
- [ ] All exports are listed in `__all__`

## Alternative Approaches Considered

### Alternative 1: Async Pipeline Processing

**Rejected**: Async processing would add complexity without significant benefit for R&D use case. Serial processing is simpler, easier to debug, and sufficient for testing scenarios.

### Alternative 2: Database-Stored Prompt Templates

**Accepted**: Database storage provides centralized management, easier updates without deployments, and better integration with the evaluation system. The `prompt_versions` table already exists in the schema, making this a natural fit.

### Alternative 3: Multiple Embedding Models

**Rejected**: Using different models for chunks and queries would break vector similarity search. Single model approach is simpler and correct.

### Alternative 4: Streaming LLM Responses

**Rejected**: Streaming adds complexity and isn't necessary for R&D testing. Full response is sufficient and simpler to implement.

### Alternative 5: Hybrid Retrieval (Vector + Keyword)

**Rejected**: Out of scope for initial implementation. Vector-only retrieval is sufficient for R&D testing and can be extended later.

## Performance Considerations

### Latency Targets

- Query pipeline: < 5 seconds (p50) for typical queries
- Upload pipeline: Scales with document size (target: < 30 seconds for 10-page PDF)

### Optimization Strategies

1. **Batch Embedding Generation**: Process multiple chunks in single API call
2. **Prompt Template Caching**: Cache loaded templates in memory to avoid repeated database queries
3. **Connection Pooling**: Reuse Azure service connections and database connections

## Risk Assessment

   - **Risk**: Services may be unavailable or rate-limited
   - **Mitigation**: Retry logic with exponential backoff, clear error messages

2. **Embedding Model Compatibility**
   - **Risk**: Query and chunk embeddings must use same model
   - **Mitigation**: Enforce model consistency in configuration, validate at runtime

3. **Index Management**
   - **Risk**: Azure AI Search index may not exist or have wrong schema
   - **Mitigation**: Automatic index creation with idempotent checks, schema versioning in code, lightweight schema design

4. **Prompt Template Errors**
   - **Risk**: Missing prompt versions in database or malformed templates
   - **Mitigation**: Validate prompt version exists in database before use, provide clear error messages, cache valid templates

5. **Determinism Challenges**
   - **Risk**: LLM generation is inherently non-deterministic
   - **Mitigation**: Accept non-determinism in generation, ensure all other steps are deterministic

### Mitigation Strategies

- Comprehensive error handling at each component
- Retry logic for transient failures
- Clear error messages for debugging
- Validation at component boundaries
- Unit tests catch regressions early
