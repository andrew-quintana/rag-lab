# RFC 002 — Codebase Consolidation & Production Readiness

## Context

This RFC defines the technical approach for consolidating the codebase structure, eliminating duplication, and ensuring production readiness for the worker-queue architecture implemented in Initiative 001.

**Status**: The current Azure Functions deployment includes duplicate backend code and fragmented configuration. This RFC provides the technical design for consolidation and production readiness validation.

## Design Overview

The consolidation approach focuses on three main areas:
1. **Code Duplication Elimination**: Remove duplicate code and ensure functions import from source
2. **Configuration Consolidation**: Unify environment variable and configuration management
3. **Test Infrastructure Consolidation**: Standardize test execution and utilities

## Technical Design

### 1. Code Duplication Elimination

#### 1.1 Remove Duplicate Code
- Delete `infra/azure/azure_functions/backend/rag_eval/` directory
- Update `.gitignore` to ensure it's not accidentally committed

#### 1.2 Update Function Entry Points
- Modify all 4 function `__init__.py` files to import from project root
- Use path manipulation to add project root to `sys.path` before imports
- Ensure dotenv loading happens before path manipulation

**Example Pattern:**
```python
from pathlib import Path
import sys
from dotenv import load_dotenv

# Load .env.local from project root
project_root = Path(__file__).parent.parent.parent.parent.parent
env_file = project_root / ".env.local"
if env_file.exists():
    load_dotenv(env_file, override=True)

# Add project root to path for backend imports
if str(project_root) not in sys.path:
    sys.path.insert(0, str(project_root))

# Now import from backend
from rag_eval.services.workers.ingestion_worker import process_ingestion_message
```

#### 1.3 Simplify Build Script
- Remove code copying logic from `build.sh`
- Build script should only verify prerequisites and prepare deployment package
- No need to copy backend code since functions import from source

### 2. Configuration Consolidation

#### 2.1 Environment Variable Loading Strategy
- **Local Development**: `.env.local` loaded via dotenv in function entry points
- **Azure Functions Runtime**: Environment variables from Function App settings
- **Precedence**: Azure settings > `.env.local` (when both present)

#### 2.2 Configuration Files
- **`local.settings.json`**: Only Azurite connection string and runtime settings
- **`.env.local`**: All application configuration (database, Azure AI services, etc.)
- **Azure Function App Settings**: Production configuration (secrets, endpoints, etc.)

#### 2.3 Documentation
- Document configuration precedence clearly
- Provide examples for local and cloud configuration
- Document migration path from current to consolidated structure

### 3. Test Infrastructure Consolidation

#### 3.1 Test Execution Unification
- Create unified test execution scripts for local and cloud
- Use pytest markers (`@pytest.mark.local`, `@pytest.mark.cloud`) consistently
- Register markers in `pytest.ini` or `conftest.py` to eliminate warnings

#### 3.2 Test Fixtures Consolidation
- Consolidate duplicate test fixtures
- Create shared fixtures for common test setup (database, queues, etc.)
- Standardize test utilities and helpers

#### 3.3 Test Scripts
- `scripts/test_functions_local.sh`: Run local Phase 5 tests
- `scripts/test_functions_cloud.sh`: Run cloud Phase 5 tests
- `scripts/test_functions_all.sh`: Run all tests with appropriate markers

### 4. Deployment Process Optimization

#### 4.1 Build Script Simplification
- Remove backend code copying
- Verify function entry points are correct
- Validate requirements.txt is present
- Prepare deployment package without code duplication

#### 4.2 Deployment Verification
- Verify functions can import from project root in Azure environment
- Test that path resolution works correctly in cloud
- Validate environment variable loading in production

## Implementation Phases

### Phase 1: Code Duplication Elimination
1. Update function entry points to import from project root
2. Remove duplicate backend code directory
3. Test local function execution with new import structure
4. Update build script to remove copying logic

### Phase 2: Configuration Consolidation
1. Document configuration loading strategy
2. Standardize `.env.local` and `local.settings.json` usage
3. Update documentation with configuration precedence
4. Test configuration loading in local and cloud environments

### Phase 3: Test Infrastructure Consolidation
1. Register pytest markers
2. Consolidate test fixtures and utilities
3. Create unified test execution scripts
4. Run Phase 5 tests with consolidated infrastructure

### Phase 4: Production Readiness Validation
1. Complete Phase 5 local testing
2. Complete Phase 5 cloud testing
3. Resolve all identified issues
4. Validate performance requirements
5. Verify production deployment stability

## Risks & Mitigations

### Risk 1: Path Resolution Issues in Azure
**Mitigation**: Test path resolution thoroughly in Azure environment before removing duplicate code. Have rollback plan to restore copying if needed.

### Risk 2: Import Errors in Production
**Mitigation**: Comprehensive testing in staging environment. Monitor function logs closely after deployment.

### Risk 3: Configuration Precedence Confusion
**Mitigation**: Clear documentation and examples. Validation scripts to check configuration.

### Risk 4: Test Infrastructure Breaking Changes
**Mitigation**: Incremental consolidation with testing at each step. Maintain backward compatibility where possible.
