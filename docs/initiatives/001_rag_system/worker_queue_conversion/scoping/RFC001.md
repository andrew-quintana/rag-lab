# RFC 001 — RAG Ingestion Worker–Queue Architecture Conversion

## Context

This RFC defines the technical architecture and implementation plan for converting the synchronous RAG ingestion pipeline to an asynchronous worker–queue architecture on Azure, as specified in [PRD001.md](./PRD001.md) and [context.md](./context.md).

**Status**: The current ingestion pipeline processes documents synchronously in a single request. This RFC provides the technical design for converting to an asynchronous worker–queue architecture.

## Design Overview

The worker–queue architecture converts the synchronous ingestion pipeline into four independent workers, each triggered by Azure Storage Queue messages. The architecture maintains all existing service modules unchanged, wrapping them in workers that add load/persist operations for intermediate data.

### Pipeline Flow

1. **Upload → Ingestion Queue**: Upload endpoint enqueues message to `ingestion-uploads`
2. **Ingestion Worker**: Downloads file → calls `extract_text_from_document()` → persists extracted text → enqueues to `ingestion-chunking`
3. **Chunking Worker**: Loads extracted text → calls `chunk_text()` → persists chunks → enqueues to `ingestion-embeddings`
4. **Embedding Worker**: Loads chunks → calls `generate_embeddings()` → persists embeddings → enqueues to `ingestion-indexing`
5. **Indexing Worker**: Loads chunks + embeddings → calls `index_chunks()` → updates final status

Each stage uses its own queue for modularity and failure isolation.

## Technical Architecture

### Queue Infrastructure

#### Azure Storage Queues

Use **Azure Storage Queues** (backed by the same storage account as Blob storage) for simplicity.

**Queue Names:**
- `ingestion-uploads` — triggers ingestion worker
- `ingestion-chunking` — triggers chunking worker
- `ingestion-embeddings` — triggers embedding worker
- `ingestion-indexing` — triggers indexing worker
- `ingestion-dead-letter` — for poison messages / unrecoverable errors (optional but recommended)

**Queue Configuration:**
- Use Azure Storage Account Queue Storage service
- Messages are small JSON payloads referencing persisted data via IDs, not raw bytes
- Queue visibility timeout: 30 seconds (default)
- Message TTL: 7 days (default)

#### Message Schema

Consistent JSON payload structure across all queues:

```json
{
  "document_id": "doc_123",
  "source_storage": "azure_blob|supabase",
  "filename": "policy.pdf",
  "attempt": 1,
  "stage": "uploaded|parsed|chunked|embedded|indexed",
  "metadata": {
    "tenant_id": "tenant_abc",
    "user_id": "user_123",
    "mime_type": "application/pdf"
  }
}
```

**Field Descriptions:**
- `document_id` (string, required): Primary key used throughout pipeline
- `source_storage` (enum, required): `"azure_blob"` or `"supabase"` - tells worker which storage backend to use
- `filename` (string, required): Original filename for logging/debugging
- `attempt` (integer, required): Incremented on retries; used to decide when to dead-letter
- `stage` (enum, required): Current processing stage for debugging & metrics: `"uploaded"`, `"parsed"`, `"chunked"`, `"embedded"`, `"indexed"`
- `metadata` (object, optional): Additional metadata including `tenant_id`, `user_id`, `mime_type`

### Worker Architecture

Each worker is a separate Azure Function with a single responsibility. Workers follow a **Load → Process → Persist** pattern:

1. **Load**: Retrieve input data from persistence layer
2. **Process**: Call existing service module function (unchanged)
3. **Persist**: Store output data to persistence layer
4. **Enqueue**: Send message to next queue stage
5. **Update Status**: Update document status in database

#### Ingestion Worker (Parsing)

**Trigger:** Azure Function with queue trigger for `ingestion-uploads` queue

**Processing Steps:**
1. Parse queue message and extract `document_id`, `source_storage`
2. **Load**: Resolve file location from `document_id` + `source_storage`
   - If `source_storage == "supabase"`: Call `download_document_from_storage(document_id, config)`
   - If `source_storage == "azure_blob"`: Call Azure Blob Storage APIs
3. **Process**: Call existing `extract_text_from_document(file_content: bytes, config) -> str` function
   - This function internally handles 2-page batching and free-tier limits via `_batch_extract_pages` and `_extract_page_range`
   - Function is unchanged - workers wrap it
4. **Persist**: Store extracted text in persistence layer (see Persistence Infrastructure section)
   - Keyed by `document_id` for retrieval by chunking worker
5. Update `documents` table: `status = 'parsed'`, `parsed_at = now()`
6. Enqueue message to `ingestion-chunking` queue with `stage = "parsed"`

**Error Handling:**
- Retry transient Azure errors (network, rate limits) with exponential backoff
- After N attempts (e.g., 3-5), send to `ingestion-dead-letter` queue
- Mark document `status = 'failed_parsing'` on permanent failure

**Idempotency:**
- Check `documents.status` before processing
- If status is already `'parsed'` or beyond, no-op and log warning

#### Chunking Worker

**Trigger:** Azure Function with queue trigger for `ingestion-chunking` queue

**Processing Steps:**
1. Parse queue message and extract `document_id`
2. **Load**: Retrieve extracted text for `document_id` from persistence layer
3. **Process**: Call existing `chunk_text(text: str, config, document_id: str, chunk_size: int, overlap: int) -> List[Chunk]` function
   - Uses deterministic fixed-size config
   - Function is unchanged - workers wrap it
4. **Persist**: Store chunks in persistence layer (see Persistence Infrastructure section)
   - Store with `document_id` and `chunk_id` for retrieval by embedding worker
5. Update `documents.status = 'chunked'`
6. Enqueue message to `ingestion-embeddings` queue (single message per document, not per chunk)

**Error Handling:**
- Most failures will be data/validation issues; fail fast
- Mark `status = 'failed_chunking'` and optionally dead-letter

**Idempotency:**
- Check `documents.status` before processing
- If status is already `'chunked'` or beyond, no-op and log warning
- Chunking worker should be safe if called twice (either overwrite or detect existing chunks for `document_id`)

#### Embedding Worker

**Trigger:** Azure Function with queue trigger for `ingestion-embeddings` queue

**Processing Steps:**
1. Parse queue message and extract `document_id`
2. **Load**: Fetch all chunks for `document_id` from persistence layer
3. **Process**: Call existing `generate_embeddings(chunks: List[Chunk], config) -> List[List[float]]` function
   - Function handles batching + retries against Azure AI Foundry
   - Function is unchanged - workers wrap it
4. **Persist**: Store embeddings in persistence layer (see Persistence Infrastructure section)
   - Store alongside chunks or in separate table keyed by `chunk_id`
5. Update `documents.status = 'embedded'`
6. Enqueue message to `ingestion-indexing` queue

**Error Handling:**
- Retry transient Azure Foundry failures with backoff (function already has retry wrapper)
- On repeated failures, mark `status = 'failed_embeddings'` and dead-letter

**Idempotency:**
- Check `documents.status` before processing
- If status is already `'embedded'` or beyond, no-op and log warning

#### Indexing Worker

**Trigger:** Azure Function with queue trigger for `ingestion-indexing` queue

**Processing Steps:**
1. Parse queue message and extract `document_id`
2. **Load**: Retrieve chunks + embeddings for `document_id` from persistence layer
3. **Process**: Call existing `index_chunks(chunks: List[Chunk], embeddings: List[List[float]], config) -> None` function
   - Function handles partial failures using existing logic
   - Function is unchanged - workers wrap it
4. Handle partial failures (some docs failing to index) using existing logic
5. Update `documents.status = 'indexed'`, `indexed_at = now()`

**Error Handling:**
- Retry Azure Search transient errors via existing `_retry_with_backoff` function
- After N attempts, mark `status = 'failed_indexing'` and dead-letter

**Idempotency:**
- Check `documents.status` before processing
- If status is already `'indexed'`, no-op and log warning
- Indexing can be repeated safely: `upload_documents` in Azure Search is already idempotent per document ID

### Persistence Infrastructure

Workers require persistence of intermediate data between pipeline stages. The current synchronous pipeline passes data in-memory, but workers need to load from and persist to storage.

#### Extracted Text Storage

**Requirement:** Persist extracted text after ingestion worker completes, keyed by `document_id` for retrieval by chunking worker.

**Storage Options:**
1. **Database Column**: Add `extracted_text TEXT` column to `documents` table
   - Pros: Simple, single query to retrieve
   - Cons: Large text may impact database performance, TEXT column size limits
2. **Supabase Storage / Azure Blob Storage**: Store as file keyed by `document_id`
   - Pros: No database size impact, supports arbitrary length
   - Cons: Additional storage service dependency

**Recommendation:** Start with database column for simplicity; migrate to storage if size becomes an issue.

#### Chunks Storage

**Requirement:** Persist chunks after chunking worker completes, support retrieval of all chunks for a `document_id` by embedding worker, and queryable by `chunk_id` for indexing worker.

**Storage Options:**
1. **Database Table**: Create `chunks` table with columns:
   ```sql
   CREATE TABLE chunks (
     chunk_id VARCHAR(255) PRIMARY KEY,
     document_id UUID NOT NULL REFERENCES documents(id),
     text TEXT NOT NULL,
     metadata JSONB,
     created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   );
   CREATE INDEX idx_chunks_document_id ON chunks(document_id);
   ```
   - Pros: Relational integrity, easy queries, supports metadata
   - Cons: Database size growth with many chunks
2. **JSON Storage**: Store chunks as JSON array in `documents` table or separate storage
   - Pros: Simple, single document
   - Cons: Less queryable, harder to update individual chunks

**Recommendation:** Use database table for queryability and metadata support.

#### Embeddings Storage

**Requirement:** Persist embeddings after embedding worker completes, support retrieval of embeddings alongside chunks by indexing worker.

**Storage Options:**
1. **Column in Chunks Table**: Add `embedding` column to `chunks` table (JSONB or vector type)
   ```sql
   ALTER TABLE chunks ADD COLUMN embedding JSONB;
   ```
   - Pros: Single query to get chunk + embedding, maintains relationship
   - Cons: Large JSONB columns, database size growth
2. **Separate Embeddings Table**: Create `embeddings` table:
   ```sql
   CREATE TABLE embeddings (
     chunk_id VARCHAR(255) PRIMARY KEY REFERENCES chunks(chunk_id),
     embedding_vector JSONB NOT NULL,
     dimension INTEGER NOT NULL,
     created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   );
   ```
   - Pros: Separates concerns, easier to manage large vectors
   - Cons: Requires JOIN for retrieval

**Recommendation:** Use column in chunks table for simplicity; migrate to separate table if needed.

#### Document Status Management

**Requirement:** Track document status through pipeline stages with status transitions and timestamps.

**Database Schema Changes:**
```sql
-- Add status column if not exists (may already exist)
ALTER TABLE documents ADD COLUMN IF NOT EXISTS status VARCHAR(50) DEFAULT 'uploaded';

-- Add timestamp columns for each stage
ALTER TABLE documents ADD COLUMN IF NOT EXISTS parsed_at TIMESTAMP;
ALTER TABLE documents ADD COLUMN IF NOT EXISTS chunked_at TIMESTAMP;
ALTER TABLE documents ADD COLUMN IF NOT EXISTS embedded_at TIMESTAMP;
ALTER TABLE documents ADD COLUMN IF NOT EXISTS indexed_at TIMESTAMP;

-- Create index for status queries
CREATE INDEX IF NOT EXISTS idx_documents_status ON documents(status);
```

**Status State Machine:**
- `uploaded` — Document uploaded, ready for processing
- `processing` — Currently being processed (optional intermediate state)
- `parsed` — Text extraction complete
- `chunked` — Chunking complete
- `embedded` — Embedding generation complete
- `indexed` — Indexing complete (final success state)
- `failed_parsing` — Ingestion worker failed
- `failed_chunking` — Chunking worker failed
- `failed_embeddings` — Embedding worker failed
- `failed_indexing` — Indexing worker failed

**Idempotency Implementation:**
- Workers check `documents.status` before processing
- If status is already equal to or beyond worker's stage, no-op and log warning
- Status checks are atomic (database transaction)

**Azure Functions** (Consumption Plan)
- 1 Function App hosting four functions:
  - `ingestion-worker` (queue trigger: `ingestion-uploads`)
  - `chunking-worker` (queue trigger: `ingestion-chunking`)
  - `embedding-worker` (queue trigger: `ingestion-embeddings`)
  - `indexing-worker` (queue trigger: `ingestion-indexing`)

**Alternative:** Azure Container Apps with queue-listening services, but Functions are leanest for this use case.

**Azure Storage Account**
- **Blob containers** (if using Azure for raw document storage): e.g., `documents`
- **Queue Storage**:
  - `ingestion-uploads`
  - `ingestion-chunking`
  - `ingestion-embeddings`
  - `ingestion-indexing`
  - `ingestion-dead-letter` (optional, but recommended)

#### Cognitive & AI Services

**Azure AI Document Intelligence**
- Used by `ingestion.py` via `DocumentIntelligenceClient` and `prebuilt-read` model
- Free-tier constraints: 2 pages per request (handled by existing `_batch_extract_pages`)

**Azure AI Foundry (or Azure OpenAI)**
- Deployed model: `"text-embedding-3-small"` for embeddings

**Azure AI Search**
- Single Search service with index defined and managed via `_ensure_index_exists` in `search.py`

**Azure Application Insights**
- For metrics, traces, and log centralization from all workers

**Azure Key Vault**
- Store sensitive config:
  - Azure Document Intelligence keys
  - Azure AI Foundry API keys
  - Azure AI Search keys
  - (Optionally) Supabase URL + key

## Migration Strategy

1. **Introduce queues and workers** *without* turning off the current synchronous ingestion path
2. For new uploads, prefer enqueuing a message into `ingestion-uploads` instead of calling the whole pipeline synchronously
3. Gradually move UI / API endpoints to rely on document `status` and show users ingestion progress
4. Once stable, remove or deprecate any direct "do everything in one request" ingestion paths

**Migration Timeline:**
- Week 1-2: Persistence infrastructure and queue setup
- Week 3-4: Worker implementation
- Week 5: API integration and testing
- Week 6: Parallel run (both synchronous and asynchronous)
- Week 7: Full cutover to asynchronous path

## Risks & Mitigation

### Technical Risks

1. **Azure Functions Cold Start Latency**
   - **Risk**: Cold starts may affect worker response time
   - **Mitigation**: Use Azure Functions Premium plan for critical workers, or implement warm-up strategies

2. **Queue Message Ordering**
   - **Risk**: Azure Storage Queues don't guarantee message ordering
   - **Mitigation**: Workers check document status before processing (idempotency) to handle out-of-order messages safely

3. **Existing Service Module Compatibility**
   - **Risk**: Wrapping existing modules may reveal compatibility issues
   - **Mitigation**: Comprehensive testing of existing modules in worker context, maintain module interfaces unchanged

4. **Persistence Layer Performance**
   - **Risk**: Database queries for large chunks/embeddings may be slow
   - **Mitigation**: Use indexes, consider pagination, monitor query performance

5. **Database Connection Pooling**
   - **Risk**: Multiple workers may exhaust connection pools
   - **Mitigation**: Implement connection pooling, monitor connection usage

## Alternative Approaches Considered

### Alternative 1: Azure Service Bus
- **Pros**: Better ordering guarantees, more features
- **Cons**: More expensive, overkill for this use case
- **Decision**: Use Azure Storage Queues for simplicity and cost

### Alternative 2: Durable Functions
- **Pros**: Built-in orchestration, state management
- **Cons**: More complex, vendor lock-in, learning curve
- **Decision**: Use simple queue triggers for transparency and simplicity

### Alternative 3: Single Worker with Internal Queuing
- **Pros**: Simpler deployment
- **Cons**: No failure isolation, can't scale stages independently
- **Decision**: Use separate workers for modularity and failure isolation
