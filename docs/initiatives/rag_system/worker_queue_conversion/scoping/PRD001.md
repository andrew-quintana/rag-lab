# PRD 001 — RAG Ingestion Worker–Queue Architecture Conversion

## Problem & Goals

### Problem Statement
The current RAG ingestion pipeline processes documents synchronously in a single request, causing several critical issues:

1. **Blocking User Experience**: Users must wait for the entire ingestion pipeline (upload → parsing → chunking → embedding → indexing) to complete before receiving a response, which can take minutes for large documents.

2. **No Failure Isolation**: If any stage fails (e.g., embedding generation), the entire pipeline fails and must be restarted from the beginning, wasting previous processing work.

3. **Limited Scalability**: Synchronous processing prevents parallel processing of multiple documents and cannot leverage Azure's serverless compute capabilities for independent scaling of each stage.

4. **Poor Observability**: It's difficult to track progress through the pipeline stages, debug failures at specific stages, or retry failed stages independently.

5. **Resource Inefficiency**: Long-running synchronous requests tie up server resources and prevent efficient resource utilization across different pipeline stages.

6. **Azure Free-Tier Constraints**: The current synchronous approach makes it challenging to maintain constant throughput while respecting Azure Document Intelligence free-tier limits (2 pages per request) without blocking the entire pipeline.

### Goals
The Worker–Queue Architecture Conversion must accomplish:

1. **Asynchronous Processing**
   - Convert synchronous ingestion pipeline to asynchronous worker–queue architecture
   - Enable non-blocking document uploads with immediate response
   - Allow users to track document processing status through pipeline stages

2. **Modular Pipeline Stages**
   - Implement independent workers for each pipeline stage (ingestion, chunking, embedding, indexing)
   - Enable failure isolation so one stage failure doesn't require restarting the entire pipeline
   - Support independent scaling and resource allocation per stage

3. **Preserve Existing Functionality**
   - Keep all existing service modules (`ingestion.py`, `chunking.py`, `embeddings.py`, `search.py`, etc.) unchanged
   - Wrap existing modules in workers without reimplementing core logic
   - **Add persistence layers** for intermediate data (extracted text, chunks, embeddings) to enable worker independence
   - Workers will load data from storage, call existing functions, then persist results
   - Maintain compatibility with Azure Document Intelligence free-tier constraints (2-page batching)

4. **Query-Time Pipeline Unchanged**
   - Keep query-time RAG pipeline (`pipeline.py`) completely unchanged
   - Ensure ingestion refactor does not break retrieval or generation functionality

5. **Idempotency & Reliability**
   - Implement idempotent workers that can safely retry operations
   - Support dead-letter queues for poison messages
   - Enable graceful error handling and retry logic with exponential backoff

6. **Observability & Monitoring**
   - Track document status through each pipeline stage
   - Enable debugging and monitoring of individual pipeline stages
   - Support metrics and logging for each worker

### Impact on Design Choices
- **Existing Service Modules**: Must preserve all existing service modules and wrap them in workers rather than reimplementing
- **Azure Document Intelligence Constraints**: Worker design must respect 2-page batching for free-tier while maintaining throughput
- **Storage Backend Flexibility**: Workers must support both Supabase Storage and Azure Blob Storage backends
- **Query Pipeline Isolation**: Ingestion refactor must not affect query-time RAG pipeline functionality
- **Supabase Schema**: Requires adding persistence infrastructure for intermediate data:
  - Extracted text storage (database column or separate storage)
  - Chunks table/storage (to persist chunks between chunking and embedding stages)
  - Embeddings storage (to persist embeddings between embedding and indexing stages)
  - Status and timestamp columns to `documents` table for pipeline tracking
- **Azure Resource Group**: Must integrate with existing Azure resources (Storage Account, AI services) and add new resources (Functions, Queues)

## User Stories & Acceptance Criteria

### User Story 1: Asynchronous Document Upload
**As a** user uploading documents  
**I want to** receive an immediate response after uploading a document  
**So that** I don't have to wait for the entire processing pipeline to complete

**Acceptance Criteria:**
- [ ] Upload endpoint returns immediately with `document_id` and initial status (`uploaded`)
- [ ] Document is enqueued to `ingestion-uploads` queue for processing
- [ ] User can query document status via API to track processing progress
- [ ] Status transitions: `uploaded` → `processing` → `parsed` → `chunked` → `embedded` → `indexed`
- [ ] Failed stages show appropriate error status (`failed_parsing`, `failed_chunking`, etc.)

### User Story 2: Independent Pipeline Stage Processing
**As a** system operator  
**I want** each pipeline stage to run independently in separate workers  
**So that** failures in one stage don't require restarting the entire pipeline

**Acceptance Criteria:**
- [ ] Ingestion worker processes documents independently from chunking worker
- [ ] Chunking worker processes independently from embedding worker
- [ ] Embedding worker processes independently from indexing worker
- [ ] Each worker has its own queue (`ingestion-uploads`, `ingestion-chunking`, `ingestion-embeddings`, `ingestion-indexing`)
- [ ] Workers can be scaled independently based on queue depth
- [ ] Failed stages can be retried without reprocessing successful stages

### User Story 3: Document Status Tracking
**As a** user or system operator  
**I want to** track document processing status through each pipeline stage  
**So that** I can monitor progress and debug issues

**Acceptance Criteria:**
- [ ] Document status is updated in database after each pipeline stage completes
- [ ] Status includes timestamps for each stage (`parsed_at`, `chunked_at`, `embedded_at`, `indexed_at`)
- [ ] API endpoint allows querying document status by `document_id`
- [ ] Status transitions are logged for observability
- [ ] Failed stages show error details in status or metadata

### User Story 4: Idempotent Worker Operations
**As a** system operator  
**I want** workers to be idempotent  
**So that** duplicate messages or retries don't cause data corruption or duplicate processing

**Acceptance Criteria:**
- [ ] Workers check document status before processing and skip if already completed
- [ ] Chunking worker safely handles duplicate messages (overwrites or detects existing chunks)
- [ ] Indexing worker uses idempotent Azure Search operations
- [ ] Workers log warnings when skipping already-completed stages
- [ ] Retry logic doesn't cause duplicate chunks or embeddings

### User Story 5: Error Handling and Dead-Letter Queue
**As a** system operator  
**I want** failed messages to be handled gracefully with retry logic and dead-letter queues  
**So that** transient failures are automatically retried and permanent failures are isolated

**Acceptance Criteria:**
- [ ] Workers retry transient Azure errors (network, rate limits) with exponential backoff
- [ ] After N retry attempts, messages are sent to `ingestion-dead-letter` queue
- [ ] Document status is updated to appropriate failure state (`failed_parsing`, `failed_chunking`, etc.)
- [ ] Dead-letter queue messages can be manually inspected and reprocessed
- [ ] Retry attempts are tracked in message metadata (`attempt` field)

### User Story 6: Azure Document Intelligence Free-Tier Support
**As a** system operator  
**I want** the ingestion worker to respect Azure Document Intelligence free-tier constraints (2 pages per request)  
**So that** the system works within free-tier limits while maintaining throughput

**Acceptance Criteria:**
- [ ] Ingestion worker uses existing `_batch_extract_pages` and `_extract_page_range` functions
- [ ] Multi-page documents are processed in 2-page batches
- [ ] Batch processing doesn't block the entire pipeline
- [ ] Throughput is maintained through parallel batch processing
- [ ] Free-tier constraints are handled transparently to other pipeline stages

## Functional Requirements

### FR1: Queue-Based Pipeline Architecture
- Implement queues for each pipeline stage to enable asynchronous processing
- Messages should contain document identifiers and processing metadata
- Support dead-letter queues for failed messages
- Enable message passing between pipeline stages

### FR2: Pipeline Stage Workers
- **Ingestion Worker**: Extract text from documents using existing ingestion module
- **Chunking Worker**: Chunk extracted text using existing chunking module
- **Embedding Worker**: Generate embeddings for chunks using existing embedding module
- **Indexing Worker**: Index chunks and embeddings using existing search module
- Each worker processes independently and can be scaled separately
- Workers wrap existing service modules without reimplementing core logic

### FR3: Intermediate Data Persistence
- Persist extracted text between ingestion and chunking stages
- Persist chunks between chunking and embedding stages
- Persist embeddings between embedding and indexing stages
- Enable workers to load data from previous stages
- Persistence must support retrieval by document identifier

### FR4: Document Status Tracking
- Track document status through each pipeline stage
- Support status queries via API
- Enable failure status tracking per stage
- Support idempotent operations (workers can safely retry)

### FR5: Upload Endpoint Modification
- Modify upload endpoint to enqueue messages instead of processing synchronously
- Return immediately with document identifier and initial status
- Add status query endpoint to track document processing progress
- Optionally maintain synchronous path for backward compatibility during migration

## Non-Functional Requirements

### NFR1: Performance

- **Latency**: Upload endpoint should return within 1 second (no processing wait)
- **Throughput**: System should maintain constant throughput despite Azure Document Intelligence free-tier constraints
- **Parallel Processing**: Multiple documents can be processed in parallel across pipeline stages
- **Queue Processing**: Workers should process messages within 30 seconds of enqueue (under normal load)

### NFR2: Reliability

- **Fault Tolerance**: Single stage failure doesn't require restarting entire pipeline
- **Retry Logic**: Transient failures automatically retried with exponential backoff (max 3-5 attempts)
- **Dead-Letter Handling**: Permanent failures isolated to dead-letter queue for manual inspection
- **Idempotency**: All workers must be idempotent to handle duplicate messages safely

### NFR3: Scalability

- **Independent Scaling**: Each worker can be scaled independently based on queue depth
- **Azure Functions Consumption Plan**: Workers use serverless consumption plan for automatic scaling
- **Queue Depth Monitoring**: System should handle queue depths of 100+ messages per queue
- **Resource Efficiency**: Workers only consume resources when processing messages

### NFR4: Security

- **Secret Management**: Azure Key Vault integration for sensitive config (API keys, connection strings)
- **Access Control**: Workers use managed identities or service principals for Azure resource access
- **Data Isolation**: Multi-tenant support via `tenant_id` in message metadata
- **Secure Storage**: Document storage access controlled via existing Supabase/Azure Blob security

### NFR5: Maintainability

- **Code Reuse**: Existing service modules (`ingestion.py`, `chunking.py`, etc.) remain unchanged and can be wrapped as-is
- **Worker Wrapping**: Workers wrap existing functions by adding load/persist operations, not reimplementing core logic
- **Persistence Layer**: Persistence infrastructure for intermediate data is separate from core service modules for maintainability
- **Modularity**: Each worker is a separate deployment for independent updates
- **Documentation**: Clear documentation of worker responsibilities, message flows, and persistence layer design

### NFR6: Observability

- **Logging**: All workers log processing steps, errors, and status updates
- **Metrics**: Azure Application Insights integration for metrics, traces, and log centralization
- **Status Tracking**: Document status visible through API and database
- **Error Visibility**: Failed stages show clear error messages and can be debugged independently

### NFR7: Cost Efficiency

- **Serverless Compute**: Azure Functions Consumption Plan for pay-per-use pricing
- **Queue Storage**: Azure Storage Queues are cost-effective for message queuing
- **Resource Optimization**: Workers only run when processing messages (no idle costs)
- **Azure Free-Tier**: Respect Azure Document Intelligence free-tier constraints to avoid costs

### NFR8: Local Development

- **Local Testing**: Workers can be tested locally using Azure Storage Emulator or local queue simulation
- **Development Experience**: Clear setup instructions for local worker development
- **Debugging**: Local workers can be debugged with standard Python debugging tools
- **Configuration**: Environment-based configuration supports local and Azure environments

### NFR9: Determinism

- **Reproducible Processing**: Same document processed through pipeline produces same results
- **Idempotent Operations**: Re-processing same document doesn't create duplicates
- **Consistent Chunking**: Chunking remains deterministic (existing `chunk_text` function)
- **Status Consistency**: Status updates are atomic and consistent

### NFR10: Modularity

- **Component Isolation**: Each worker is isolated and can be updated independently
- **Queue Decoupling**: Stages communicate only through queues (no direct dependencies)
- **Service Module Independence**: Existing service modules remain independent and testable
- **Failure Isolation**: Failures in one worker don't affect other workers

## Risks / Open Questions

### Technical Risks

1. **Azure Functions Cold Start Latency**
   - **Risk**: Azure Functions may have cold start delays affecting worker response time
   - **Mitigation**: Use Azure Functions Premium plan for critical workers, or implement warm-up strategies

2. **Queue Message Ordering**
   - **Risk**: Azure Storage Queues don't guarantee message ordering, which may cause issues if messages processed out of order
   - **Mitigation**: Workers check document status before processing (idempotency) to handle out-of-order messages safely

3. **Existing Service Module Compatibility**
   - **Risk**: Wrapping existing service modules in workers may reveal compatibility issues or assumptions about synchronous execution
   - **Mitigation**: Comprehensive testing of existing modules in worker context, maintain module interfaces unchanged

4. **Azure Document Intelligence Rate Limiting**
   - **Risk**: Free-tier rate limits may cause ingestion worker bottlenecks
   - **Mitigation**: Implement proper retry logic with exponential backoff, monitor queue depth, scale ingestion workers if needed

5. **Database Connection Pooling**
   - **Risk**: Multiple workers accessing Supabase database may exhaust connection pools
   - **Mitigation**: Implement connection pooling, monitor connection usage, use connection string configuration

6. **Message Schema Evolution**
   - **Risk**: Future changes to message schema may break existing workers
   - **Mitigation**: Version message schema, implement backward compatibility, comprehensive schema validation

### Open Questions

1. **Migration Strategy**
   - Should synchronous upload path be deprecated immediately or maintained in parallel?
   - How long should both paths coexist before removing synchronous path?
   - Should there be a feature flag to switch between synchronous and asynchronous paths?

2. **Storage Backend Selection**
   - Should workers support both Supabase Storage and Azure Blob Storage, or standardize on one?
   - How should `source_storage` be determined (user selection, automatic detection, configuration)?

3. **Extracted Text Storage**
   - Where should extracted text be stored (database, Supabase Storage, Azure Blob Storage)?
   - What is the storage format (raw text, compressed, encrypted)?

4. **Chunk Storage**
   - Where should chunks be stored (database table, document store, separate storage)?
   - Should chunks be stored separately from embeddings or together?

5. **Worker Deployment**
   - Should all workers be in a single Azure Function App or separate Function Apps?
   - How should worker configuration be managed (environment variables, Key Vault, separate config files)?

6. **Monitoring and Alerting**
   - What metrics should be monitored (queue depth, worker processing time, failure rates)?
   - What alerting thresholds should be set (queue depth > X, failure rate > Y)?

7. **Dead-Letter Queue Handling**
   - How should dead-letter queue messages be handled (manual inspection, automated retry, notification)?
   - Should there be a UI or API for inspecting and reprocessing dead-letter messages?

## Assumptions & Dependencies

### Assumptions

1. **Existing Service Modules Stable**: Existing service modules (`ingestion.py`, `chunking.py`, `embeddings.py`, `search.py`) are stable and don't require changes for worker integration - they can be wrapped as-is
2. **Persistence Infrastructure Required**: While core service modules can be wrapped unchanged, persistence infrastructure must be added for intermediate data (extracted text, chunks, embeddings) to enable worker independence
3. **Azure Services Available**: All required Azure services (Functions, Storage, Document Intelligence, AI Foundry, AI Search) are available and configured
4. **Supabase Database Access**: Workers have network access to Supabase database and storage, and database schema can be extended for persistence needs
5. **Document Size Limits**: Documents are reasonably sized (< 100MB) to fit in memory and queue message metadata
6. **Free-Tier Constraints**: Azure Document Intelligence free-tier constraints (2 pages per request) will continue to be respected
7. **Query Pipeline Unchanged**: Query-time RAG pipeline will not be modified during this initiative
8. **Migration Timeline**: Migration can be done gradually without requiring immediate cutover from synchronous to asynchronous processing

### Dependencies

1. **Azure Functions Runtime**: Azure Functions Python runtime (v2 or v4) for worker deployment
2. **Azure Storage Account**: Existing or new Storage Account with Queue Storage enabled
3. **Azure Key Vault**: For secure storage of API keys and connection strings (recommended)
4. **Azure Application Insights**: For monitoring and logging (recommended)
5. **Supabase Database**: Existing Supabase database with `documents` table - **requires schema additions** for persistence:
   - Extracted text storage (column or separate table)
   - Chunks table (if not using separate storage)
   - Embeddings storage (column in chunks table or separate table)
   - Status and timestamp columns for pipeline tracking
6. **Supabase Storage or Azure Blob Storage**: For document file storage (and optionally for intermediate data storage)
7. **Azure Document Intelligence**: Existing Azure Document Intelligence resource with free-tier or paid tier
8. **Azure AI Foundry**: Existing Azure AI Foundry resource with embedding model deployed
9. **Azure AI Search**: Existing Azure AI Search service with index configured
10. **Python Dependencies**: Existing Python dependencies for service modules (Azure SDK, Supabase client, etc.)
