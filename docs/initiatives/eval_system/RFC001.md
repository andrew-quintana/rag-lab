# RFC 001 — RAG Evaluation MVP (LLM-as-Judge + BEIR + Meta-Eval)

## Context

This RFC defines the technical architecture and implementation plan for the RAG Evaluation MVP system, as specified in [PRD001.md](./PRD001.md). The evaluation system provides a quantitative framework to measure how retrieval method changes and prompt changes affect downstream RAG system performance through LLM-as-Judge evaluation, meta-evaluation for judge reliability, and BEIR-style retrieval metrics.

**Status**: The evaluation system is currently not implemented. This RFC provides the technical design for implementing the complete evaluation stack.

## Design Overview

The RAG Evaluation MVP consists of a modular evaluation pipeline that integrates with the existing RAG system to provide quantitative metrics for retrieval performance, output correctness, and judge reliability. The system is designed for:

- **Determinism**: Reproducible evaluation results using deterministic fixtures
- **Modularity**: Isolated evaluation components that can be tested independently
- **Simplicity**: MVP-simple design with no dashboard, minimal dataset, local execution
- **Extensibility**: Easy to extend with additional evaluation metrics or judge strategies

### Component Architecture

The evaluation system consists of the following modular components:

1. **Evaluation Dataset Generator**: Creates synthetic QA pairs from test fixtures
2. **LLM-as-Judge Orchestrator**: Deterministic Python script that orchestrates three LLM nodes
3. **Hallucination LLM-Node**: Binary classification for hallucination detection
4. **Hallucination Cost LLM-Node**: Cost type classification (-1 for opportunity cost, +1 for resource cost)
5. **Hallucination Impact LLM-Node**: Impact magnitude calculation (0-3 scale)
6. **Meta-Evaluator**: Validates judge verdicts for reliability
7. **BEIR Metrics Evaluator**: Computes recall@k, precision@k, nDCG@k

#### LLM-as-Judge Orchestrator

```python
def evaluate_answer_with_judge(
    query: str,
    retrieved_context: List[RetrievalResult],
    model_answer: str,
    reference_answer: str,
    config: Optional[Config] = None
) -> JudgeEvaluationResult:
    """
    Evaluate model answer using LLM-as-Judge with deterministic orchestration.
    
    Orchestrates evaluation process:
    1. Correctness classification (direct comparison to reference answer)
    2. Hallucination binary classification (grounding analysis against retrieved context)
    3. Cost type classification (if hallucination detected)
    4. Impact magnitude calculation (if hallucination detected)
    
    Args:
        query: Original user query
        retrieved_context: List of retrieved chunks with similarity scores
        model_answer: Generated answer from RAG system
        reference_answer: Gold reference answer for comparison
        config: Application configuration (optional, defaults to Config.from_env())

        JudgeEvaluationResult object containing:
        - correctness_binary: bool (true if answer is correct)
        - hallucination_binary: bool (true if hallucination detected)
        - hallucination_cost: Optional[int] (-1 for opportunity cost, +1 for resource cost, None if no hallucination)
        - hallucination_impact: Optional[float] (0-3 scale, None if no hallucination)
        - reasoning: str (reasoning trace from all LLM node outputs)
        - failure_mode: Optional[str] (e.g., "cost misstatement", "omitted deductible")
        
    Raises:
        AzureServiceError: If Azure Foundry LLM calls fail
        ValueError: If inputs are invalid or empty
    """
```

#### Meta-Evaluator (Deterministic Validation)

```python
def meta_evaluate_judge(
    judge_output: JudgeEvaluationResult,
    retrieved_context: List[RetrievalResult],
    model_answer: str,
    reference_answer: str,
    extracted_costs: Optional[Dict[str, Any]] = None,  # From cost extraction LLM-node
    actual_costs: Optional[Dict[str, Any]] = None  # From cost extraction LLM-node on chunks

    """
    Deterministically evaluate whether LLM-as-Judge's verdicts are correct.
    
    Uses rule-based validation logic to compare judge output against ground truth.
    No LLM calls - pure deterministic function.
    
    Validation checks:
    1. correctness_binary: Compare model answer to reference answer (exact or semantic similarity)
    2. hallucination_binary: Check if model answer claims are supported by retrieved chunks
    3. hallucination_cost: Validate cost direction against extracted costs (if available)
    4. hallucination_impact: Validate impact magnitude against cost differences (if available)
    
    Args:
        judge_output: JudgeEvaluationResult from LLM-as-Judge
        retrieved_context: Original retrieved chunks (ground truth for hallucination detection)
        model_answer: Original model answer evaluated by judge
        reference_answer: Original reference answer (ground truth for correctness)
        extracted_costs: Optional cost information extracted from model answer
        actual_costs: Optional cost information extracted from retrieved chunks
        
    Returns:
        MetaEvaluationResult object containing:
        - judge_correct: bool (True if all judge verdicts are correct, False otherwise)
        - explanation: Optional[str] (Deterministic explanation of validation results)
        
    Raises:
        ValueError: If inputs are invalid
    """
```

### Phase 6: LLM-as-Judge Orchestrator

**Component**: `rag_eval/services/evaluator/judge.py`

**Implementation Details:**
- Implement deterministic Python script orchestrator for LLM-as-Judge
- Sequential execution with conditional branching:
  1. Call correctness LLM-node (always)
  2. Call hallucination LLM-node (always)
  3. If hallucination detected: call hallucination_cost LLM-node
  4. If hallucination detected: extract costs using cost extraction LLM-node, then call hallucination_impact LLM-node
- Construct reasoning trace from all LLM node outputs
- Assemble final JudgeEvaluationResult with all fields including correctness
- Handle edge cases: zero chunks, empty answers, LLM failures

**Key Functions:**
```python
def evaluate_answer_with_judge(query: str, retrieved_context: List[RetrievalResult], model_answer: str, reference_answer: str, config: Optional[Config] = None) -> JudgeEvaluationResult
def _orchestrate_judge_evaluation(query: str, retrieved_context: List[RetrievalResult], model_answer: str, reference_answer: str, config: Config) -> JudgeEvaluationResult
def _construct_reasoning_trace(correctness_result: bool, hallucination_result: bool, cost_result: Optional[int], impact_result: Optional[float], node_reasonings: List[str]) -> str
```

**Orchestration Logic:**
```python
# Step 1: Correctness classification
correctness_binary = classify_correctness(query, model_answer, reference_answer, config)

# Step 2: Hallucination classification
hallucination_binary = classify_hallucination(retrieved_context, model_answer, config)

# Step 3: Cost classification (if hallucination detected)
hallucination_cost = None
if hallucination_binary:
    hallucination_cost = classify_hallucination_cost(model_answer, retrieved_context, config)

# Step 4: Impact calculation (if hallucination detected)
hallucination_impact = None
if hallucination_binary:
    # Extract costs from model answer using cost extraction LLM-node
    model_answer_cost = extract_costs(model_answer, config)
    
    # Extract costs from retrieved chunks using cost extraction LLM-node
    chunks_text = " ".join([chunk.chunk_text for chunk in retrieved_context])
    actual_cost = extract_costs(chunks_text, config)
    
    # Calculate impact using extracted costs
    hallucination_impact = calculate_hallucination_impact(model_answer_cost, actual_cost, config)

# Step 5: Construct reasoning trace and assemble result
reasoning = _construct_reasoning_trace(...)
```

**Dependencies:**
- Phases 2, 3, 4, 5 (correctness, hallucination, cost classification, cost extraction, impact LLM nodes)
- Config for API credentials

### Phase 7: Meta-Evaluator (Deterministic Validation)

**Component**: `rag_eval/services/evaluator/meta_eval.py`

**Implementation Details:**
- Implement deterministic meta-evaluator to validate judge verdicts
- Pure Python function with no LLM calls
- Use rule-based validation logic to compare judge output against ground truth
- Validate correctness_binary by comparing model answer to reference answer
- Validate hallucination_binary by checking if model answer is grounded in retrieved chunks
- Validate hallucination_cost and hallucination_impact using extracted costs (if available)
- Return binary classification: judge_correct or judge_incorrect
- Provide deterministic explanation of validation results

**Key Functions:**
```python
def meta_evaluate_judge(judge_output: JudgeEvaluationResult, retrieved_context: List[RetrievalResult], model_answer: str, reference_answer: str, extracted_costs: Optional[Dict[str, Any]] = None, actual_costs: Optional[Dict[str, Any]] = None) -> MetaEvaluationResult
def _validate_correctness(judge_correctness: bool, model_answer: str, reference_answer: str) -> bool
def _validate_hallucination(judge_hallucination: bool, model_answer: str, retrieved_context: List[RetrievalResult]) -> bool
def _validate_cost_classification(judge_cost: Optional[int], extracted_costs: Dict[str, Any], actual_costs: Dict[str, Any]) -> bool
def _validate_impact_magnitude(judge_impact: Optional[float], extracted_costs: Dict[str, Any], actual_costs: Dict[str, Any]) -> bool
def _generate_explanation(validation_results: Dict[str, bool]) -> str
```

**Validation Logic:**
1. **Correctness Validation**: 
   - If judge says `correctness_binary: true`, check if model answer matches reference answer (exact or high semantic similarity)
   - If judge says `correctness_binary: false`, check if model answer differs from reference answer
   
2. **Hallucination Validation**:
   - If judge says `hallucination_binary: true`, check if model answer contains claims not supported by retrieved chunks
   - If judge says `hallucination_binary: false`, check if all model answer claims are supported by retrieved chunks
   
3. **Cost Classification Validation** (if costs available):
   - Compare extracted costs vs actual costs to determine expected cost direction
   - Validate judge's `hallucination_cost` matches expected direction
   
4. **Impact Magnitude Validation** (if costs available):
   - Calculate expected impact magnitude based on cost differences
   - Validate judge's `hallucination_impact` is within reasonable range of expected impact

**Dependencies:**
- JudgeEvaluationResult from Phase 6
- Optional: Extracted costs from Phase 4 (cost extraction LLM-node)
- No Azure Foundry API calls required

**Component**: `rag_eval/services/evaluator/beir_metrics.py`

**Implementation Details:**
- Implement BEIR-style retrieval metrics computation
- Compute recall@k, precision@k, nDCG@k for retrieval results
- Use ground-truth passage IDs from evaluation dataset

**Key Functions:**
```python
def compute_beir_metrics(retrieved_chunks: List[RetrievalResult], ground_truth_chunk_ids: List[str], k: int = 5) -> BEIRMetricsResult
def _compute_recall_at_k(retrieved_chunk_ids: List[str], ground_truth_chunk_ids: List[str], k: int) -> float
def _compute_precision_at_k(retrieved_chunk_ids: List[str], ground_truth_chunk_ids: List[str], k: int) -> float
def _compute_ndcg_at_k(retrieved_chunks: List[RetrievalResult], ground_truth_chunk_ids: List[str], k: int) -> float
```

**Metric Formulas:**
- **Recall@k**: (Number of relevant chunks in top-k) / (Total number of relevant chunks)
- **Precision@k**: (Number of relevant chunks in top-k) / k
- **nDCG@k**: Normalized discounted cumulative gain, using relevance scores (1 for relevant, 0 for irrelevant)

**Dependencies:**

## Technical Decisions

### Decision 1: Deterministic Script Orchestrator for LLM-as-Judge

**Decision**: Use a deterministic Python script orchestrator (not an LLM supervisor) to coordinate the three-step evaluation process.

**Rationale**:
- Simpler architecture: no need for complex LLM supervisor logic
- Clear control flow: conditional branching based on hallucination_binary result
- Easier to test and debug: deterministic execution order
- Lower cost: no additional LLM call for orchestration
- Sufficient for MVP: simple sequential execution meets requirements

**Implementation**: The orchestrator uses standard Python if/else logic to conditionally invoke cost and impact LLM nodes only when hallucination is detected.

### Decision 2: Separate LLM Nodes for Cost and Impact Classification

**Decision**: Use separate LLM nodes for hallucination cost classification and impact calculation, rather than combining them.

**Rationale**:
- Clear separation of concerns: cost direction vs. impact magnitude are distinct analyses
- Easier to test and validate each component independently
- More modular: can swap or improve individual nodes without affecting others
- Better error handling: failures in one node don't prevent the other from running
- Follows single responsibility principle

**Implementation**: Three separate LLM nodes (hallucination, cost, impact) with clear interfaces and independent execution.

### Decision 3: LLM-Based Impact Calculation (Not Deterministic Function)

**Decision**: Use an LLM node (not a deterministic function) for hallucination impact calculation.

**Rationale**:
- Handles mixed resource types: time, money, and steps require nuanced reasoning about relative importance
- Qualitative assessments: non-quantitative hallucinations need impact assessment
- Context-aware: considers real-world consequences beyond simple numeric differences
- Flexible: can adapt to different types of hallucinations and failure modes

**Implementation**: `hallucination_impact` LLM node uses GPT-4o-mini to analyze cost differences and compute impact magnitude on 0-3 scale.

### Decision 4: Azure Foundry GPT-4o-mini for All LLM Calls

**Decision**: Use Azure Foundry GPT-4o-mini for all evaluation LLM calls (judge, correctness, hallucination, cost classification, cost extraction, and impact nodes). Meta-evaluator is deterministic and does not require LLM calls.

**Rationale**:
- Consistency: matches RAG system LLM provider
- Cost efficiency: GPT-4o-mini is sufficient for evaluation tasks
- Simplicity: single provider reduces configuration complexity
- Integration: leverages existing Azure Foundry setup and credentials

**Implementation**: All LLM calls use `config.azure_ai_foundry_*` settings with GPT-4o-mini model.

### Decision 5: Local Execution with Optional Database Logging

**Decision**: Evaluation runs entirely locally, with optional Supabase logging for results.

**Rationale**:
- MVP simplicity: local execution is easier to reason about and debug
- No external dependencies required: evaluation can run without database
- Optional logging: supports future analytics without blocking MVP
- Reproducibility: local execution ensures deterministic results

**Implementation**: Evaluation pipeline runs locally, with optional `log_evaluation_result()` function for database persistence.

### Decision 6: Fixture-Based Evaluation Dataset

**Decision**: Generate evaluation dataset from test fixtures (`healthguard_select_ppo_plan.pdf`) and store in `backend/tests/fixtures/`.

**Rationale**:
- Reproducibility: version-controlled fixtures ensure consistent evaluation
- Simplicity: single document is sufficient for MVP
- Integration: leverages existing test infrastructure
- Extensibility: can expand to multiple documents in future phases

**Implementation**: Dataset generator creates synthetic QA pairs from PDF fixture and stores in `backend/tests/fixtures/evaluation_dataset/`.

### Decision 7: BEIR Metrics as Pure Python Functions

**Decision**: Implement BEIR metrics (recall@k, precision@k, nDCG@k) as pure Python functions without LLM calls.

**Rationale**:
- Deterministic: standard information retrieval metrics are well-defined
- Fast: no LLM latency for metric computation
- Accurate: mathematical formulas ensure correct calculations
- Standard: BEIR metrics are industry-standard retrieval evaluation metrics

**Implementation**: Pure Python functions compute metrics using ground-truth chunk IDs and retrieved rankings.

### Decision 8: Deterministic Meta-Evaluator (No LLM)

**Decision**: Implement meta-evaluator as a deterministic Python function that validates judge verdicts against ground truth, rather than using an LLM.

**Rationale**:
- Reliability: Deterministic validation against ground truth is more reliable than LLM-based evaluation
- Cost efficiency: No additional LLM calls for meta-evaluation
- Speed: Deterministic functions are faster than LLM calls
- Reproducibility: Deterministic validation produces consistent results
- Simplicity: Rule-based validation logic is easier to understand and debug
- Ground truth available: We have reference answers and retrieved chunks to validate against

**Implementation**: Meta-evaluator uses rule-based logic to:
- Validate `correctness_binary` by comparing model answer to reference answer
- Validate `hallucination_binary` by checking if model answer is grounded in retrieved chunks
- Validate `hallucination_cost` and `hallucination_impact` using extracted costs (if available)

### Decision 9: Temperature=0.1 for All LLM Calls

**Decision**: Use temperature=0.1 for all evaluation LLM calls to maximize reproducibility.

**Rationale**:
- Reproducibility: lower temperature reduces variance in LLM outputs
- Consistency: enables comparison across evaluation runs
- Sufficient quality: GPT-4o-mini with low temperature still produces good evaluations
- Experimental use case: R&D testing benefits from consistent results

**Implementation**: All Azure Foundry LLM calls use `temperature=0.1` parameter.

## Alternative Approaches Considered

### Alternative 1: LLM Supervisor for Judge Orchestration

**Rejected**: Using an LLM supervisor to orchestrate judge evaluation would add complexity and cost without significant benefit. Deterministic script orchestrator is simpler and sufficient for MVP.

### Alternative 2: Combined Cost and Impact LLM Node

**Rejected**: Combining cost classification and impact calculation into a single LLM node would reduce modularity and make testing more difficult. Separate nodes provide better separation of concerns.

### Alternative 3: Deterministic Function for Impact Calculation

**Rejected**: Pure numeric calculation cannot handle mixed resource types (time/money/steps) and qualitative impact assessments. LLM node provides necessary reasoning capability.

### Alternative 4: Multiple LLM Providers

**Rejected**: Using different providers (OpenAI, Anthropic) would add complexity and inconsistency. Azure Foundry provides sufficient quality and consistency with existing RAG system.

### Alternative 5: Real-Time Evaluation Dashboard

**Rejected**: Dashboard functionality is explicitly out of scope for MVP. Programmatic evaluation APIs are sufficient for R&D use case.

### Alternative 7: Multi-Document Evaluation Dataset

**Rejected**: Single-document dataset is sufficient for MVP. Multi-document expansion can be added in future phases.

## Package Structure Standards

### Overview

All evaluation packages in the `rag_eval` codebase must follow standardized package structure conventions to ensure proper imports, testability, and maintainability.

### Requirements

#### 1. `__init__.py` Files

- **All packages must have `__init__.py` files**: Every directory that is a Python package must contain an `__init__.py` file with proper exports.

- **Export Pattern**: Each `__init__.py` file should:
  - Include a docstring describing the package/module purpose
  - Export commonly used classes/functions using `from ... import ...` statements
  - Define an `__all__` list for explicit exports

#### 2. Import Conventions

- **Absolute Imports Preferred**: Always use absolute imports from the package root:
  ```python
  from rag_eval.core.interfaces import ModelAnswer, RetrievalResult
  from rag_eval.services.evaluator.judge import evaluate_answer_with_judge
  ```

- **No `sys.path` Manipulation**: Production code must not manipulate `sys.path`.

#### 3. Package Organization

- **Core Primitives** (`rag_eval/core/`): Interfaces, exceptions, configuration, logging
- **Services** (`rag_eval/services/evaluator/`): Evaluation components organized by function
  - `judge.py`: LLM-as-Judge orchestrator
  - `hallucination.py`: Hallucination LLM-node
  - `hallucination_cost.py`: Cost classification LLM-node
  - `hallucination_impact.py`: Impact calculation LLM-node
  - `meta_eval.py`: Meta-evaluator
  - `beir_metrics.py`: BEIR metrics evaluator
  - `orchestrator.py`: Evaluation pipeline orchestrator
  - Note: Evaluation dataset is a JSON file in `backend/tests/fixtures/evaluation_dataset/validation_dataset.json`, not a Python module
  - `logging.py`: Optional evaluation result logging

#### 4. Testing Configuration

- **pytest Configuration**: Tests should be in `backend/tests/test_evaluator_*.py`
- **Test Organization**: Tests organized by component (test_judge.py, test_hallucination.py, etc.)
