# PRD 001 — RAG Evaluation MVP (LLM-as-Judge + BEIR + Meta-Eval)

## Problem & Goals

### Problem Statement
AI engineers experimenting with RAG systems need a quantitative evaluation framework to measure how retrieval method changes and prompt changes affect downstream performance. Currently, there is no systematic way to evaluate RAG system outputs for correctness, hallucination detection, or retrieval quality. This evaluation stack provides an experimental harness that combines LLM-as-Judge evaluation, meta-evaluation for judge reliability, and BEIR-style retrieval metrics to enable data-driven RAG system improvements.

### Goals
The RAG Evaluation MVP must accomplish:

1. **Quantitative Evaluation Metrics**
   - Provide measurable metrics for retrieval performance (BEIR-style: recall@k, precision@k, nDCG@k)
   - Provide measurable metrics for output correctness (LLM-as-Judge with hallucination detection)
   - Provide measurable metrics for judge reliability (Meta-Evaluator)

2. **Hallucination Detection & Impact Assessment**
   - Detect hallucinations in model answers using LLM-as-Judge with deterministic orchestration
   - Classify hallucination cost types (opportunity cost vs. resource cost)
   - Calculate hallucination impact magnitude (0-3 scale) for real-world consequence assessment
   - Tag failure modes (copay, coinsurance, deductible, out-of-pocket max, eligibility errors)

3. **Experimental Simplicity**
   - MVP-simple design: no dashboard, minimal dataset, Azure stack only where necessary
   - Everything runs locally and is easy to reason about
   - Clear boundaries between components (retriever, RAG generator, judge, meta-judge, BEIR evaluator)

4. **Reproducibility**
   - Deterministic fixtures for consistent evaluation
   - All example inputs inside `backend/tests/fixtures/...`
   - Reproducible evaluation runs for comparison across RAG system variations

5. **Modularity**
   - Each evaluation component is isolated and testable independently:
     - LLM-as-Judge (deterministic script orchestrator)
     - Hallucination LLM-node (binary classification)
     - Hallucination cost LLM-node (cost type classification)
     - Hallucination impact LLM-node (impact magnitude calculation)
     - Meta-Evaluator
     - BEIR metrics evaluator

## Functional Requirements

### FR1: LLM-as-Judge (Deterministic Script Orchestrator)

1. **Deterministic Orchestration Implementation**
   - Implement deterministic Python script to orchestrate evaluation process
   - Sequential execution with conditional branching:
     1. Correctness classification (via correctness LLM-node)
     2. Hallucination classification (via hallucination LLM-node)
     3. Cost type classification (via hallucination_cost LLM-node, if hallucination detected)
     4. Impact calculation (via hallucination_impact LLM-node, if hallucination detected)
   - No LLM supervisor needed - simple conditional logic based on hallucination_binary result

2. **Input Processing**
   - Accept query as input
   - Accept retrieved context (chunks) as input
   - Accept model answer as input
   - Accept reference answer as input
   - Validate all inputs are non-empty

3. **Output Schema**
   - Return structured JSON with:
     - `correctness_binary`: true | false
     - `hallucination_binary`: true | false
     - `hallucination_cost`: -1 | 1 (only if hallucination_binary: true)
     - `hallucination_impact`: number [0, 3] (only if hallucination_binary: true)
     - `reasoning`: string (reasoning trace constructed from LLM node outputs)
     - `failure_mode`: string (optional, e.g., "cost misstatement", "omitted deductible")

4. **LLM Node Invocation**
   - Call correctness LLM-node (always)
   - Call hallucination LLM-node (always)
   - Call hallucination_cost LLM-node when hallucination_binary: true
   - Call hallucination_impact LLM-node when hallucination_binary: true
   - Construct reasoning trace from outputs of all invoked LLM nodes

### FR2: Correctness LLM-Node (Direct Comparison)

1. **Classification Logic**
   - Directly compare model answer to gold reference answer
   - Assess whether model answer is correct or incorrect
   - Return binary classification

2. **Input Processing**
   - Accept query
   - Accept model answer
   - Accept gold reference answer

3. **Output**
   - Return `correctness_binary`: true | false

### FR3: Hallucination LLM-Node (Binary Classification)

1. **Classification Logic**
   - Analyze model answer against retrieved evidence only (not reference answer)
   - Perform strict grounding analysis: does model answer contain information not supported by evidence?
   - Return binary classification: true (hallucination present) or false (no hallucination)

2. **Input Processing**
   - Accept retrieved context (chunks)
   - Accept model answer

3. **Output**
   - Return `hallucination_binary`: true | false

### FR4: Hallucination Cost LLM-Node (Cost Type Classification)

1. **Classification Logic**
   - Analyze whether hallucination overestimates or underestimates costs (time/money/steps)
   - Compare model answer cost to retrieved chunks (ground truth) only (not reference answer)
   - Classify cost impact direction:
     - **-1 = Opportunity Cost**: Model overestimated cost, potentially dissuading user from seeking care
     - **+1 = Resource Cost**: Model underestimated cost, potentially persuading user to pursue care

2. **Input Processing**
   - Accept model answer
   - Accept retrieved chunks (for ground-truth context)

3. **Output**
   - Return `hallucination_cost`: -1 | 1
   - Classification applies to all hallucinations (quantitative and non-quantitative)

4. **Invocation**
   - Invoked by deterministic script orchestrator when `hallucination_binary: true`

### FR5: Hallucination Impact LLM-Node (Impact Magnitude Calculation)

1. **Impact Calculation Logic**
   - Analyze difference between model answer cost and actual cost from chunks
   - Consider mixed resource types (time, money, steps) and their relative importance
   - Assess magnitude of real-world consequence across different dimensions
   - Compute scaling factor [0, 3] representing impact magnitude

2. **Input Processing**
   - Accept cost in time/money/steps from model answer
   - Accept actual cost in time/money/steps from retrieved chunks (ground truth)

3. **Output**
   - Return `hallucination_impact`: number [0, 3]
   - Serves as scaling factor combined with hallucination_cost (-1 or +1) for total consequence

4. **Rationale**
   - Uses LLM node (not deterministic function) because it must handle mixed resource types and assess relative importance, requiring nuanced reasoning

5. **Invocation**
   - Invoked by deterministic script orchestrator when `hallucination_binary: true`

### FR6: Meta-Evaluator (Deterministic Validation)

1. **Evaluation Logic**
   - Deterministically evaluate whether LLM-as-Judge's verdicts are correct
   - Compare judge output against ground truth (reference answer, retrieved chunks)
   - Use deterministic classification and regression evaluation (no LLM calls)
   - Validate correctness_binary by comparing model answer to reference answer
   - Validate hallucination_binary by checking if model answer is grounded in retrieved chunks
   - Validate hallucination_cost and hallucination_impact by comparing extracted costs

2. **Input Processing**
   - Accept judge output (JudgeEvaluationResult with all fields)
   - Accept model answer
   - Accept reference answer (ground truth for correctness)
   - Accept retrieved context (ground truth for hallucination detection)

3. **Output**
   - Return `judge_correct`: true | false
   - Return optional concise explanation (deterministic, not LLM-generated)

4. **Implementation**
   - Pure Python deterministic function (no LLM calls)
   - Uses rule-based validation logic

### FR7: BEIR-Style Retrieval Metrics

1. **Metric Computation**
   - Compute recall@k for retrieval results
   - Compute precision@k for retrieval results
   - Compute nDCG@k for retrieval results
   - Use ground-truth passage IDs from evaluation dataset

2. **Integration**
   - Compute metrics for each evaluation example
   - Include beir_failure_scale_factor in evaluation dataset for context-aware hallucination impact judging

3. **Dataset Format**
   - Follow BEIR dataset format with query-passage pairs
   - Include relevance labels for ground-truth passages

### FR8: Evaluation Dataset Construction (Development Task)

1. **Dataset Creation**
   - **Development Task**: Coding agent creates a small validation dataset (5 samples) for system validation
   - Source: `backend/tests/fixtures/sample_documents/healthguard_select_ppo_plan.pdf`
   - Create 5 QA pairs manually (not via automated generation function)
   - Each sample includes gold reference answers and ground-truth chunks
   - **Note**: Full evaluation dataset will be created by humans later; this is only for validation

2. **Dataset Schema**
   - Each example includes:
     - Question
     - Gold reference answer
     - Ground-truth chunks/paragraphs for retrieval relevance
     - beir_failure_scale_factor (float, representing retrieval challenge/severity)

3. **Storage**
   - Store 5 validation samples in `backend/tests/fixtures/evaluation_dataset/` for version control
   - Use deterministic format (JSON) for reproducibility
   - Format: `validation_dataset.json` with list of EvaluationExample objects

### FR9: Evaluation Pipeline Integration

1. **Pipeline Orchestration**
   - Integrate with existing RAG system components:
     - `retrieve(query, k)` returning list of (chunk, score)

   - Execute evaluation pipeline: Retrieval → RAG generation → LLM-as-Judge → Meta-Eval → Metrics

2. **API Design**

## Risks / Open Questions

### Technical Risks

1. **LLM Non-Determinism**
   - **Risk**: GPT-4o-mini outputs may vary across calls, causing inconsistent evaluation results
   - **Mitigation**: Use low temperature (0.1) for judge calls, document expected variance, use meta-evaluator to catch judge errors

2. **Hallucination Cost Classification Ambiguity**
   - **Risk**: Ambiguous cases where it's unclear if cost was overestimated (opportunity cost) or underestimated (resource cost)
   - **Mitigation**: Provide clear guidelines in hallucination_cost LLM-node prompt, use meta-evaluator to validate classifications

3. **Quantitative Hallucination Impact Calculation**
   - **Risk**: Financial cost differences may not be computable from chunks for all cases
   - **Mitigation**: Use LLM reasoning (hallucination_impact LLM-node) to handle mixed resource types and qualitative assessments

4. **Judge False Positives/Negatives**
   - **Risk**: Judge may incorrectly mark grounded answers as hallucinations or miss actual hallucinations
   - **Mitigation**: Implement meta-evaluator to catch judge errors, use strict grounding analysis in hallucination LLM-node

5. **Zero Relevant Passages Retrieved**
   - **Risk**: Retrieval may return zero relevant passages, causing evaluation failures
   - **Mitigation**: Handle edge case gracefully, mark as retrieval failure in metrics, allow evaluation to proceed with empty context

### Open Questions

1. **Failure Mode Risk Scoring**
   - Should failure modes eventually map to a numeric risk score for analytics?
   - **Decision**: Deferred to future phase

2. **Confidence Intervals**
   - Should we include confidence intervals for judge outputs (multiple samples)?
   - **Decision**: Deferred to MVP (single evaluation per example)

3. **Dataset Expansion**
   - Should the evaluation dataset grow beyond a single PDF in later versions?
   - **Decision**: Yes, but out of scope for MVP

4. **Judge Output Format**
   - Should judge outputs include confidence scores or probability distributions?
   - **Decision**: Binary classification for MVP, confidence scores deferred

5. **BEIR Metrics Aggregation**
   - How should BEIR metrics be aggregated across evaluation examples (mean, median, weighted)?
   - **Decision**: TBD in RFC phase
