# PRD 001 — In-Corpus Evaluation Dataset Generation System

## Problem & Goals

### Problem Statement
AI engineers need a systematic way to generate evaluation datasets for "In-Corpus" RAG evaluation. Currently, there is no automated system to create diverse, effective queries from indexed documents and generate complete evaluation datasets with ground truth retrieval context, system prompts, structured prompts, and RAG outputs. This system provides an end-to-end pipeline for generating evaluation datasets that can be used for comprehensive RAG system evaluation.

### Goals
The In-Corpus Evaluation Dataset Generation System must accomplish:

1. **Automated Query Generation**
   - Generate diverse, effective queries from Azure AI Search indexed documents
   - Focus on "In-Corpus" questions only (questions answerable from indexed documents)
   - Create queries that are effective for evaluating RAG systems
   - Include metadata linking queries to source chunks for ground truth retrieval

2. **Complete Evaluation Dataset Generation**
   - Generate evaluation datasets with all required fields:
     - Input queries
     - Retrieval queries (sanitized/changed if necessary)
     - Ground truth retrieved context for BEIR metrics
     - System prompts from Supabase
     - Structured prompts from Supabase (RAG-enabled, live)
     - RAG outputs
     - BEIR metrics
     - Placeholders for LLM and human evaluation fields
   - Support ground truth retrieval context for accurate BEIR evaluation

3. **System Prompt Management**
   - Support system prompts stored in Supabase with naming convention `{type}_system`
   - Ensure RAG-enabled prompts exist with proper system prompts
   - Query and retrieve system prompts dynamically

4. **Output Generation**
   - Generate RAG outputs using system prompts and structured prompts
   - Support batch processing of evaluation datasets
   - Update evaluation datasets with generated outputs

5. **Terminology Consistency**
   - Update all `risk_impact` references to `risk_magnitude` throughout codebase
   - Ensure consistent naming across interfaces, evaluator modules, and database schemas

## User Stories & Acceptance Criteria

### User Story 1: Generate Diverse Queries from Indexed Documents
**As a** AI engineer
**I want to** automatically generate diverse, effective queries from Azure AI Search indexed documents  
**So that** I can create evaluation datasets without manual query creation

**Acceptance Criteria:**
- [ ] Can generate queries from Azure AI Search embeddings/chunks
- [ ] Generated queries are diverse and effective for RAG evaluation
- [ ] All queries are "In-Corpus" (answerable from indexed documents)
- [ ] Queries include metadata linking to source chunks for ground truth
- [ ] Output saved to `evaluations/{eval_name}/in_corpus/eval_inputs.json`

### User Story 2: Generate Complete Evaluation Dataset
**As a** AI engineer  
**I want to** generate complete evaluation datasets from eval_inputs.json  
**So that** I have all required fields for comprehensive RAG evaluation

**Acceptance Criteria:**
- [ ] Can load eval_inputs.json and generate evaluation dataset
- [ ] Dataset includes input, retrieval_query, context (ground truth), system_prompt, structured_prompt
- [ ] System prompts queried from Supabase with `{type}_system` naming convention
- [ ] Structured prompts queried from Supabase (prompt_type="RAG-enabled", live=TRUE)
- [ ] Ground truth retrieval context included for BEIR metrics
- [ ] BEIR metrics computed using ground truth chunk IDs
- [ ] LLM and human evaluation fields initialized as null
- [ ] Output saved to `evaluations/{eval_name}/in_corpus/eval_dataset.json`

### User Story 3: Generate RAG Outputs for Evaluation Dataset
**As a** AI engineer  
**I want to** generate RAG outputs for evaluation dataset entries  
**So that** I can complete the evaluation dataset with actual RAG system outputs

**Acceptance Criteria:**
- [ ] Can load evaluation dataset and generate outputs for each entry
- [ ] Uses system_prompt and structured_prompt from dataset
- [ ] Uses retrieval_query for retrieval
- [ ] Uses retrieved context for generation
- [ ] Updates output field in dataset
- [ ] Saves updated dataset

### User Story 4: Manage System Prompts
**As a** AI engineer  
**I want to** create and manage system prompts in Supabase  
**So that** I can use them in evaluation dataset generation

**Acceptance Criteria:**
- [ ] Can create system prompts with naming convention `{type}_system`
- [ ] System prompts stored in `public.prompts` table
- [ ] Can query system prompts by prompt_type and name
- [ ] At least one basic system prompt exists for RAG-enabled prompts

### User Story 5: Update Terminology to risk_magnitude
**As a** AI engineer  
**I want to** have consistent terminology using `risk_magnitude` instead of `risk_impact`  
**So that** the codebase uses consistent naming

**Acceptance Criteria:**
- [ ] All `risk_impact` references updated to `risk_magnitude`
- [ ] Interfaces updated (JudgeEvaluationResult, MetaEvaluationResult, JudgePerformanceMetrics)
- [ ] Evaluator modules updated
- [ ] Database schemas updated (if applicable)
- [ ] All variable names, function parameters updated
