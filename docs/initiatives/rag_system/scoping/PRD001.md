# PRD 001 — RAG Lab (RAG System Testing Suite)

## Problem & Goals

### Problem Statement
AI engineers need a streamlined testing and evaluation platform (the "RAG Lab") to prototype, test, and iterate on RAG (Retrieval-Augmented Generation) system components. Currently, the RAG pipeline is not implemented, leaving a critical gap in the platform's R&D capabilities. The RAG Lab provides a simple, modular pipeline that allows AI engineers to experiment with different chunking strategies, embedding models, prompt versions, and retrieval approaches, while enabling testing groups to evaluate output quality. This is a research and development tool, not a production system.

### Goals
The RAG Lab must accomplish:

1. **Correctness**
   - Given a query, the system retrieves appropriate context and produces grounded answers that accurately reflect the source material, enabling AI engineers to validate component behavior.

2. **Reproducibility**
   - Chunking, embeddings, and retrieval are deterministic given the same inputs, ensuring consistent results across test runs for reliable experimentation.

3. **Traceability**
   - RAG pipeline operations are tracked internally; evaluation and meta-evaluation data is logged to Supabase Postgres for observability and analysis.

4. **Modularity**
   - Each subsystem is isolated and modular within the serialized, local process, allowing AI engineers to test individual components:
     - ingestion
     - chunking
     - embeddings
     - search
     - generation
     - logging

5. **Minimal API + Minimal Dependencies**
   - Streamlined API with essential routes (`POST /query`, `POST /upload`)
   - No unnecessary abstractions
   - Simple, transparent components for easy modification and testing

6. **Extendability**
   - The design should allow AI engineers to easily extend and test:
     - new prompt versions
     - new embedding models
     - new indexing strategies
   - But the initial version focuses on a simple, streamlined pipeline for R&D

## User Stories & Acceptance Criteria

### User Story 1: Building and Testing Document Ingestion Components
**As an** AI engineer  
**I want to** upload a document and have it automatically processed and indexed through a modular pipeline  
**So that** I can test and validate different ingestion, chunking, and embedding strategies

**Acceptance Criteria:**
- [ ] Upload pipeline accepts document files via `POST /upload` endpoint
- [ ] Documents are processed through OCR/text extraction using Azure Document Intelligence
- [ ] Extracted text is chunked using deterministic chunking algorithms (testable and modifiable)
- [ ] Chunks are embedded using Azure AI Foundry embedding models
- [ ] Embedded chunks are indexed in Azure AI Search as searchable vector documents
- [ ] Upload pipeline completes successfully and document is ready for querying
- [ ] Upload-related events are logged locally (not to Supabase) for debugging and analysis
- [ ] Each component (ingestion, chunking, embeddings, indexing) can be tested independently

### User Story 2: Testing Query Pipeline and Evaluating Outputs
**As an** AI engineer or testing group member  
**I want to** submit a natural language query and receive detailed pipeline results  
**So that** I can evaluate retrieval quality, prompt effectiveness, and answer quality for R&D purposes

**Acceptance Criteria:**
- [ ] Query endpoint accepts `POST /query` with `query` and `prompt_version` parameters
- [ ] Query is embedded using Azure AI Foundry embedding models
- [ ] Top-k relevant chunks are retrieved from Azure AI Search using vector similarity
- [ ] Prompt is constructed using prompt templates from Supabase `prompt_versions` table (e.g., version "v1", "v2")
- [ ] LLM generates answer using Azure AI Foundry (GPT-4o-mini or GPT-4o)
- [ ] Response includes:
   - final answer
   - retrieved context chunks
   - vector similarity scores
   - latency metrics
- [ ] Query and results are logged to Supabase for evaluation/telemetry and analysis
- [ ] Testing groups can use the query results to evaluate whether outputs are good

### User Story 3: Reproducible Testing and Experimentation
**As an** AI engineer  
**I want** the RAG pipeline to produce consistent, reproducible results  
**So that** I can reliably test different component configurations and compare results across experiments

**Acceptance Criteria:**
- [ ] Same document input produces identical chunks
- [ ] Same query produces identical embeddings
- [ ] Same query and index produce identical retrieval results
- [ ] Chunking algorithm is deterministic (no randomness)
- [ ] Embedding generation is deterministic (same model, same input = same output)
- [ ] Results can be reproduced across multiple test runs for fair component comparisons

### User Story 4: Modular Component Testing and Extension
**As an** AI engineer  
**I want** the RAG Lab to have modular, isolated components  
**So that** I can easily test, modify, and swap individual components (chunking strategies, embedding models, prompt versions) without affecting other parts of the pipeline

**Acceptance Criteria:**
- [ ] Each subsystem (ingestion, chunking, embeddings, search, generation, logging) is isolated
- [ ] Components can be tested independently
- [ ] New prompt versions can be added by inserting records into Supabase `prompt_versions` table
- [ ] System supports multiple prompt versions (v1, v2, etc.) for A/B testing via database records
- [ ] Architecture allows for future model swaps without major refactoring
- [ ] Components are simple and transparent, making it easy to understand and modify behavior

## Functional Requirements

### FR1: Upload Pipeline (`POST /upload`)
1. **Document Reception**
   - Accept file uploads via HTTP POST
   - Support common document formats (PDF, DOCX, etc.)
   - **Note (2025-01-27)**: Azure Blob Storage persistence has been removed from scope. Documents are processed in-memory without storage.

2. **Text Extraction**
   - Use Azure Document Intelligence for OCR and text extraction
   - Extract: OCR text, layout, tables, block/line segmentation
   - Process documents in-memory (no intermediate file storage)

3. **Chunking**
   - Apply deterministic chunking algorithms locally
   - Support size-based, sentence-based, or hybrid chunking strategies
   - Preserve metadata (source document, chunk position, etc.)

4. **Embedding Generation**
   - Generate embeddings for each chunk using Azure AI Foundry
   - Preserve chunk-embedding-metadata associations

5. **Indexing**
   - Store chunk embeddings and metadata in Azure AI Search
   - Create searchable vector documents for retrieval

### FR2: Query Pipeline (`POST /query`)
1. **Query Reception**
   - Accept natural language queries via HTTP POST
   - Accept optional `prompt_version` parameter (default: "v1")
   - Support testing scenarios where AI engineers or testing groups submit queries

2. **Query Embedding**
   - Embed query using Azure AI Foundry embedding models
   - Use same embedding model as chunks for compatibility
   - Enable testing of different embedding models

3. **Retrieval**
   - Query Azure AI Search using vector similarity search
   - Retrieve top-k relevant chunks with similarity scores
   - Return associated metadata for each chunk
   - Provide detailed retrieval information for evaluation

4. **Prompt Construction**
   - Load prompt template from Supabase `prompt_versions` table by version name (e.g., "v1", "v2")
   - Assemble prompt with:
     - user query
     - retrieved context chunks
     - system instructions from template
   - Produce final LLM-ready prompt
   - Enable easy testing of different prompt versions by querying different version names from database

5. **LLM Generation**
   - Send prompt to Azure AI Foundry (GPT-4o-mini or GPT-4o)
   - Generate grounded answer based on retrieved context
   - Return generated text
   - Enable testing of different LLM models

6. **Response Assembly**
   - Package response with:
     - final answer (for testing groups to evaluate quality)
     - retrieved chunks (for AI engineers to assess retrieval quality)
     - similarity scores (for analysis)
     - latency metrics (for performance testing)
     - query ID
     - prompt version used

### FR3: Logging and Observability
1. **Query Pipeline Logging**
   - Log to Supabase Postgres:
     - query text
     - retrieved chunks
     - model output
     - evaluation scores (if present, from testing groups)
     - metadata for observability and analysis
   - Enable AI engineers to analyze pipeline behavior and testing groups to track evaluation results

2. **Upload Pipeline Logging**
   - Log internally (not to Supabase):
     - document processing stats
     - chunking statistics
     - indexing results
   - Use local logging for debugging and component testing

### FR4: Python API Interface
- Expose `run_rag(query: str, prompt_version: str) -> dict` function
- Return structured data:
  - final answer
  - retrieved chunks
  - prompt version
  - metadata for logging

## Testing Requirements

### Overview
The RAG Lab must have a robust unit testing infrastructure that ensures component-level functionality persists across changes. All components must be thoroughly tested in isolation to catch regressions and validate general functionality when modifications are made.

### Testing Philosophy
- **Component-First Testing**: Each component must have comprehensive unit tests that validate its core functionality independently
- **Regression Prevention**: Tests must catch breaking changes when components are modified
- **Deterministic Testing**: All tests must be deterministic and reproducible
- **Isolation**: Components must be testable without dependencies on external services or other components
- **Test Persistence**: Component tests must persist as part of the codebase and run on every change

## Risks / Open Questions

### Technical Risks

1. **Azure Service Availability**
   - **Risk**: Azure services (Document Intelligence, AI Foundry, AI Search) may be unavailable
   - **Mitigation**: Implement retry strategies with exponential backoff; provide clear error messages

2. **Embedding Model Compatibility**
   - **Risk**: Query and chunk embeddings must use the same model for accurate similarity
   - **Mitigation**: Enforce model consistency in configuration; validate at runtime

3. **Chunking Quality**
   - **Risk**: Poor chunking may break context or lose important information
   - **Mitigation**: Implement and test multiple chunking strategies; allow for future improvements

4. **Prompt Template Management**
   - **Risk**: Prompt versions may become difficult to manage or missing from database
   - **Mitigation**: Use clear version naming conventions (e.g., "v1", "v2"); validate version exists before use; document prompt versioning strategy and database schema

5. **Determinism Challenges**
   - **Risk**: Some operations (e.g., LLM generation) are inherently non-deterministic
   - **Mitigation**: Accept non-determinism in LLM generation; ensure all other steps are deterministic

### Open Questions

1. **Document Format Support**
   - Which document formats should be supported initially? (PDF, DOCX, TXT, etc.)
   - What are the size limits for document uploads?

2. **Chunking Strategy**
   - What is the optimal chunk size for different document types?
   - Should chunking be configurable or fixed?

3. **Retrieval Parameters**
   - What is the optimal value for top-k retrieval?
   - Should this be configurable or fixed?

4. **Prompt Versioning**
   - How should prompt versions be managed and validated in the database?
   - What happens if an invalid prompt version is requested (not found in database)?
   - Should there be a default prompt version if none is specified?

5. **Error Handling**
   - What is the user experience when Azure services fail?
   - Should partial results be returned if some steps fail?

6. **Local Development**
   - What level of mocking is required for local development?
   - How should developers test without Azure credentials?

## Assumptions & Dependencies

### Assumptions
1. Documents are uploaded one at a time (no batch processing) - sufficient for R&D testing
2. Single document per index (no multi-document support) - streamlined for initial testing
3. Azure services are available and properly configured
4. AI engineers have valid Azure credentials and access
5. Supabase is available for logging evaluation/telemetry data
6. Prompt templates are stored in Supabase `prompt_versions` table for centralized management and easy updates without code deployments
7. System runs on a single server (no distributed processing) - simplified for testing
8. This is a testing/R&D platform, not a production system - simplicity and testability are prioritized over scale

### Dependencies
1. **Azure Blob Storage**: For document persistence (optional)
2. **Azure Document Intelligence**: For OCR and text extraction
3. **Azure AI Foundry**: For embeddings and LLM generation
4. **Azure AI Search**: For vector search and retrieval
5. **Supabase Postgres**: For evaluation/telemetry logging and prompt template storage (not for RAG pipeline operations like chunk storage)
6. **FastAPI**: For HTTP API endpoints
7. **Python 3.13+**: For backend implementation
